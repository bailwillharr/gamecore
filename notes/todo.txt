Replace mutable FrameState with immutable FrameInput, Mutable FrameOutput. Split systems between 'simulation' and 'presentation'.
	Simulation systems cannot modify FrameOutput, they only receive FrameInput as const ref parameter.
	Presentation systems receive FrameInput const ref, but also receive FrameOutput as a mutable ref parameter.



Pipeline variants
	If a given material doesn't use a normal map, for example, it shouldn't sample a normal texture in the fragment shader.
	Use multiple pipelines instead or maybe just use null descriptors.
	Either way, don't store pipeline in material, just what features are wanted, such as:
	 - use normal map
	 - constant roughness vs roughness map



Use the same ByteReader and ByteWriter classes engine-wide.
	No more NetByteReader for network stuff, and also stream usage for file stuff.
	Just standardise the serialised format used both for file and network.



Prefabs
	Prefabs should support game-defined components.
	To support this, every component now declares a 'static constexpr gc::Name NAME' field.
	Every component will also need serialisation and deserialisation functions (THAT ARE TYPE ERASED). I.e.,

	static void serialise(const void* comp, ByteWriter& writer);
	static void deserialise(ByteReader& reader, void* out);
	static constexpr size_t getSerialisedSize();

	gc::World will save references to all these functions in the registerComponent() call.
	This way, the prefab loader can easily get the gc::Name for a component, have no idea what it is, and do:

	world.deserialiseComponent(entity, comp_name, byte_reader)

	and gc::World will figure out the component index for comp_name, get the deserialise() function, and call:

	m_component_arrays[index].addComponent(entity) (which I'll modify to return a reference).

	Then it'll directly deserialise into the memory that the reference points to (taking to care avoid aliasing / alignment problems.)



Use 64 bit hash for gc::Name (4 billion combinations isn't that much, and I don't see a reason why Names can't be 8 bytes instead of 4)

	Add a World::getTransform() that doesn't check, since TransformComponents always exist (unless the entity doesn't exist!)
	But if the entity doesn't exist when calling getComponent<>, that should be UB anyway.
	
	The comment in findEntity() suggests that a Name->Entity map should just be made for optimisation purposes.
	But the the same name is allowed for different entities under different parents, so this won't work.

	m_component_names can probably be merged with m_component_arrays, likewise for m_system_names, which can be merged with m_systems.
